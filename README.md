## Features

<table>
<tr>
<td width="50%">

**ORBAT Overview**\
Canvas-style order-of-battle viewer with a compact card grid fallback for narrow viewports. Players can join available slots after login.

</td>
<td width="50%">

**Operation Scheduler**\
Create, edit, and manage operations with full recurrence support — daily, weekly, biweekly, and monthly. Restricted to admin and missionmaker roles.

</td>
</tr>
<tr>
<td width="50%">

**Template Builder**\
Design reusable squad and slot presets. Templates define structure and default markers, and load into operations via the scheduler.

</td>
<td width="50%">

**User Management**\
Role-based access (`admin`, `missionmaker`, `member`), per-user rank assignment, status tracking, and avatar uploads.

</td>
</tr>
<tr>
<td width="50%">

**Ranks**\
Admin-managed rank list with abbreviations, display order, and icon support. Seeded on first run, fully editable via the Ranks page.

</td>
<td width="50%">

**Campaigns**\
Link operations to named campaigns with modlists, default templates, and assigned missionmakers.

</td>
</tr>
<tr>
<td width="50%">

**File Uploads**\
Marker icons, avatars, and admin assets. Extension-validated, 5 MB cap, UUID-named storage under `uploads/`.

</td>
<td width="50%">

**Backup / Restore**\
Squad-level export and import from Settings. Select individual data sections (users, templates, ops, ranks, etc.) to restore independently.

</td>
</tr>
</table>

## Stack

| Layer | Technology | Notes |
|---|---|---|
| Frontend | React 18, Vite 5 | SPA, entry `src/main.jsx` |
| Backend | Node.js 20, Express 4 | Single process, serves API + `dist/` |
| Database | MySQL 8 | Connection pool via `db.js`; schema auto-initialised on startup |
| Auth | `jsonwebtoken`, `bcryptjs` | JWT, Bearer token |
| Uploads | `multer` | Disk storage, UUID filenames |
| Dev tooling | `concurrently`, `@vitejs/plugin-react` | Parallel dev server |

## Getting Started

### Prerequisites

- Node.js 20+
- MySQL 8 instance (local Docker or remote)

### Local setup

```bash
# 1. Clone the repo
git clone https://github.com/the-honorguard/TFOwebapp.git
cd TFOwebapp

# 2. Install dependencies
npm install

# 3. Copy and fill environment variables
cp .env.example .env
# Edit .env — see Environment Variables section below

# 4. Start a local MySQL instance (optional — uses Docker)
docker-compose up -d

# 5. Start dev server (frontend + backend in parallel)
npm run dev
```

On first run, `lib/dataStore.js` automatically creates all tables and seeds the default admin account and rank list.

### Available scripts

| Command | Description |
|---|---|
| `npm run dev` | Frontend + backend in parallel (development) |
| `npm run build` | Build frontend to `dist/` |
| `npm run server` | API server only |
| `npm run client` | Vite dev server only |

## Environment Variables

Copy `.env.example` to `.env` and set all values before starting.

| Variable | Required | Description |
|---|---|---|
| `DB_HOST` | yes | MySQL host (e.g. `127.0.0.1`) |
| `DB_PORT` | yes | MySQL port (default `3306`) |
| `DB_USER` | yes | MySQL user |
| `DB_PASSWORD` | yes | MySQL password |
| `DB_NAME` | yes | MySQL database name |
| `JWT_SECRET` | yes | Secret used to sign JWT tokens — use a long random string, never commit |
| `PORT` | no | HTTP port (default `3001`; injected automatically by cPanel) |
| `NODE_ENV` | no | Set to `production` in production environments |

> **Never commit a real `JWT_SECRET` or database password.** The `.env` file is gitignored. Use your hosting panel's environment variable manager in production.

## Architecture

### Runtime model

In production, a single Node.js process handles all traffic:

```
client request
      │
      ▼
  server.js (Express)
      ├── /api/*          REST API handlers
      ├── /uploads/*      Static file serving (uploaded assets)
      └── *               dist/index.html  (SPA catch-all)
```

### Data flow

- Unauthenticated clients load public data via `GET /api/public-data`
- Authenticated clients load full data via `GET /api/data` (JWT required)
- All writes call the API and update local React state on success
- `defaultOpSettings` is persisted to `localStorage` on the client

### Database

All tables are created by `lib/dataStore.js` on startup via `CREATE TABLE IF NOT EXISTS`. Schema migrations must be applied manually (see `todo.md` for pending changes). The connection pool is configured in `db.js` and shared across all repositories.

## API Reference

### Auth

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/api/login` | Authenticate — returns `{ token, user }` |
| `POST` | `/api/signup` | Register — returns `{ token, user }` |

All protected routes require `Authorization: Bearer <token>`.

### Data

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| `GET` | `/api/public-data` | — | Users, templates, ops, campaigns |
| `GET` | `/api/data` | required | Full data for authenticated user |
| `GET` | `/api/users/me` | required | Current user profile |
| `PUT` | `/api/users/me` | required | Update profile |
| `PUT` | `/api/users/me/password` | required | Change password |

### Ranks

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| `GET` | `/api/ranks` | — | List all ranks |
| `POST` | `/api/ranks` | admin | Create rank |
| `PUT` | `/api/ranks/:id` | admin | Update rank |
| `DELETE` | `/api/ranks/:id` | admin | Delete rank |

### Templates

```
POST    /api/templates
PUT     /api/templates/:id
DELETE  /api/templates/:id
POST    /api/templates/:id/duplicate
POST    /api/templates/:id/squads
PUT     /api/templates/:templateId/squads/:squadId
DELETE  /api/templates/:templateId/squads/:squadId
POST    /api/templates/:id/slots
PUT     /api/templates/:templateId/slots/:slotId
DELETE  /api/templates/:templateId/slots/:slotId
PUT     /api/templates/:templateId/squads/:squadId/slots/reorder
```

### Operations

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| `POST` | `/api/ops` | admin | Create op or recurrence |
| `PUT` | `/api/ops/:id` | admin / missionmaker | Update op |
| `DELETE` | `/api/ops/:id` | admin | Delete op |
| `POST` | `/api/ops/:id/join` | required | Sign up to a slot |
| `POST` | `/api/ops/:id/signoff` | required | Sign off from a slot |
| `PUT` | `/api/ops/:opId/slots/:slotId` | admin / missionmaker | Update slot metadata |
| `PUT` | `/api/ops/:opId/squads/:squadId` | admin / missionmaker | Update squad metadata |

### Uploads

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| `POST` | `/api/upload` | admin | General asset upload |
| `POST` | `/api/upload/custom-marker` | admin / missionmaker | Map marker icon |
| `POST` | `/api/upload/avatar` | required | User avatar |

### Backup

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| `GET` | `/api/backup/export` | admin | Export full data snapshot |
| `POST` | `/api/backup/import` | admin | Restore selected sections from snapshot |

## Deployment

Designed to run as a **single Node.js process** on cPanel-compatible shared hosting.

### cPanel configuration

| Field | Value |
|---|---|
| Node.js version | `20.x` |
| Application mode | `Production` |
| Startup file | `server.js` |

Set all environment variables in the cPanel Node.js App environment variable manager — never via a committed `.env` file on the server.

### First deployment

```bash
# 1. Build locally
npm run build

# 2. Upload all project files including dist/ via FTP
#    Upload: dist/, lib/, repositories/, src/, public/,
#            server.js, db.js, package.json, package-lock.json

# 3. Run NPM Install from cPanel Node.js App interface

# 4. Set environment variables in cPanel

# 5. Start the application
```

### Update workflow

```bash
# 1. Rebuild frontend
npm run build

# 2. Upload changed files via FTP (always include dist/)

# 3. Restart — cPanel > Node.js App > Restart
```

## Known Issues

See [`todo.md`](todo.md) for the full task list. Current blocking items:

- **Ranks icon column missing** — `icon` field is not yet persisted to the database; rank icons will appear blank until the schema migration is applied.
- **Roles backend not implemented** — the Roles page is present in the UI but has no database table, repository, or API routes behind it.
- **Hardcoded auth secret** — `JWT_SECRET` must be moved to an environment variable before any public exposure.

---

## Notes

- `uploads/` is created automatically on first run
- A default `admin` account is seeded on first start — change the password immediately via Settings
- File-based operations are not suited for heavy concurrent writes; a proper job queue or database-level locking should be considered at scale
- The `dist/` folder must be rebuilt and re-uploaded after any frontend change
