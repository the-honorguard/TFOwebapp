## Features

<table>
<tr>
<td width="50%">

**ORBAT Overview**\
Canvas-style order-of-battle viewer with squad markers, compact card grid fallback for narrow viewports, and square avatar/badge slot cards. Players can join or sign off available slots after login directly from the overview.

</td>
<td width="50%">

**Operation Scheduler**\
Create, edit, and manage operations with full recurrence support â€” daily, weekly, biweekly, and monthly. Restricted to admin and missionmaker roles. Squad metadata (LR/SR channels, markers) is editable per-op.

</td>
</tr>
<tr>
<td width="50%">

**Template Builder**\
Design reusable squad and slot presets in Flow mode (canvas with drag, connectors, auto-layout) or Form mode (grid). Templates define structure, default markers, and squad types, and load into operations via the scheduler.

</td>
<td width="50%">

**User Management**\
Role-based access (`admin`, `missionmaker`, `member`), per-user rank assignment with icons, status tracking, avatar uploads with in-app crop/zoom editor, and a signup flow.

</td>
</tr>
<tr>
<td width="50%">

**Ranks**\
Admin-managed rank list with abbreviations, display order, and icon support (seeded from the TFO mkdocs asset CDN). Fully editable via the Ranks page.

</td>
<td width="50%">

**Squad Types**\
Admin-managed squad type presets (Infantry, Armor, Medic, etc.) with SVG marker icons. Used in the template builder marker picker and auto-seeded on first run.

</td>
</tr>
<tr>
<td width="50%">

**Notifications**\
In-app notification system. Events like op sign-ups, slot changes, and admin actions generate notifications persisted in the DB and surfaced in the UI.

</td>
<td width="50%">

**Campaigns**\
Link operations to named campaigns with modlists, default templates, and assigned missionmakers.

</td>
</tr>
<tr>
<td width="50%">

**File Uploads**\
Marker icons, avatars, and admin assets. Extension-validated, 5 MB cap, UUID-named storage under `uploads/`.

</td>
<td width="50%">

**Backup / Restore**\
Full DB export and selective import from Settings. Choose individual data sections (users, templates, ops, ranks, etc.) to restore independently.

</td>
</tr>
</table>

## Stack

| Layer | Technology | Notes |
|---|---|---|
| Frontend | React 18, Vite 5 | SPA, entry `src/main.jsx` |
| Backend | Node.js 20, Express 4 | Single process, serves API + `dist/` |

| Logging | `winston` | Structured JSON logs, SSE log stream endpoint for admin |
| Dev tooling | `concurrently`, `@vitejs/plugin-react` | Parallel dev server |

## Getting Started

### Prerequisites

- Node.js 20+
- MySQL 8 instance (local Docker or remote)

### Local setup

```bash
# 1. Clone the repo
git clone https://github.com/the-honorguard/TFOwebapp.git
cd TFOwebapp

# 2. Install dependencies
npm install

# 3. Copy and fill environment variables
cp .env.example .env
# Edit .env â€” see Environment Variables section below

# 4. Start a local MySQL instance (optional â€” uses Docker)
docker-compose up -d

# 5. Start dev server (frontend + backend in parallel)
npm run dev
```

On first run, `lib/dataStore.js` automatically creates all tables, runs column migrations, and seeds the default rank list.

### Available scripts

| Command | Description |
|---|---|
| `npm run dev` | Frontend + backend in parallel (development) |
| `npm run build` | Build frontend to `dist/` |
| `npm run server` | API server only |
| `npm run client` | Vite dev server only |
| `npm test` | Run unit tests (`orbatLayout.test.js`) |

## Environment Variables

Copy `.env.example` to `.env` and set all values before starting. **Never commit `.env` to version control.**

| Variable | Required | Description |
|---|---|---|
| `DB_HOST` | yes | MySQL host (e.g. `127.0.0.1`) |
| `DB_PORT` | yes | MySQL port (default `3306`) |
| `DB_USER` | yes | MySQL username |
| `DB_PASSWORD` | yes | MySQL password |
| `DB_NAME` | yes | MySQL database name |
| `JWT_SECRET` | **yes** | Secret used to sign JWT tokens â€” use a long random string, never commit |
| `PORT` | no | HTTP port (default `3001`) |
| `NODE_ENV` | no | Set to `production` in production |
| `ENABLE_DEMO_SEED` | no | Set to `1` to seed demo op/template on first `POST /init` |

> âš ď¸Ź **Security:** `JWT_SECRET` must be a long, random, unpredictable string. Use your hosting panel's environment variable manager in production â€” never a committed `.env` file on the server.

## Architecture

### Runtime model

In production, a single Node.js process handles all traffic:

```
client request
      â”‚
      â–Ľ
  server.js (Express)
      â”śâ”€â”€ /api/*          REST API handlers
      â”śâ”€â”€ /uploads/*      Static file serving (uploaded assets)
      â”śâ”€â”€ /health         DB health check
      â””â”€â”€ *               dist/index.html  (SPA catch-all)
```

### Data flow

- Unauthenticated clients load public data via `GET /api/public-data`
- Authenticated clients load full data via `GET /api/data` (JWT required)
- All writes call the API and update local React state on success
- `defaultOpSettings` is persisted to `localStorage` on the client

### Database schema

All tables are created by `lib/dataStore.js` on startup via `CREATE TABLE IF NOT EXISTS`. Column migrations (e.g. INT â†’ BIGINT for timestamp IDs) run automatically. The connection pool is configured in `db.js` and shared across all repositories.

**Tables:** `users`, `user_profiles`, `ranks`, `templates`, `ops`, `recurrences`, `campaigns`, `modlists`, `files`, `backups`, `roles`, `squad_types`, `notifications`

## Deployment

Designed to run as a **single Node.js process** on cPanel-compatible shared hosting (tested on o2switch).

### cPanel configuration

| Field | Value |
|---|---|
| Node.js version | `20.x` |
| Application mode | `Production` |
| Startup file | `server.js` |

Set all environment variables via the cPanel Node.js App environment variable manager.

### First deployment

```bash
# 1. Build locally
npm run build

# 2. Upload via FTP â€” keep same folder structure:
#    dist/, lib/, repositories/, scripts/, public/,
#    server.js, db.js, package.json, package-lock.json

# 3. Run NPM Install from cPanel Node.js App interface

# 4. Set environment variables in cPanel

# 5. Start the application
```

> **Note:** `tfo-emoji.png` must be in `public/` locally so Vite copies it to `dist/` root during build.

### Update workflow

```bash
# 1. Rebuild frontend
npm run build

# 2. Upload changed files via FTP (always include dist/ and any changed server files)

# 3. Restart â€” cPanel > Node.js App > Restart
```

### Automated deploy (Windows / o2switch)

From Windows, the complete build, upload and Passenger restart flow can be run with:

```powershell
.\scripts\deploy-o2switch.ps1
```

The script preserves the remote `.env`, `uploads/`, and `logs/` directories. Pass `-EnvFile` only when intentionally replacing the production environment file. Use `-IdentityFile "$HOME\.ssh\o2switch"` after configuring an SSH key.

See [`todo.md`](todo.md) for the full active task list, priorities, and acceptance criteria. Key open items:

- **Hardcoded auth secret** â€” `JWT_SECRET` must be set via environment variable; default fallback should be removed
- **Duplicate route conflict** â€” two `PUT` handlers exist for the same section endpoint; the second silently overrides the first
- Schema migrations for production DBs must be run manually when new columns are added (e.g. `ALTER TABLE ranks ADD COLUMN icon VARCHAR(1024)`)

## Notes

- `uploads/` and `logs/` are created automatically on first run
- The `dist/` folder must be rebuilt and re-uploaded after any frontend change
- Static files in `public/` (logo, markers) are copied to `dist/` at build time â€” do not reference them from `src/` via `import`
- The DB schema check runs automatically on server startup â€” new tables are created without manual intervention
